把真實的家政生科教室裡那盞日光燈關掉一半，你伸手就會在桌面投出一塊清楚的黑影。但開刀房裡那盞罩著醫生團隊的大燈,卻好像怎麼擋都擋不出影子——這不是錯覺,而是一整套光學設計的結果。

![無影燈光學模擬器介面:學習目標、光路逆行演示、2D 幾何光路追跡](/assets/blog/lamp-raytrace.png)

我做了一個小而專注的互動模擬器,專門回答一個問題:**為什麼外科手術無影燈幾乎沒有影子?** 這篇文章會把它整個拆開來:核心光學原理、我畫的射線示意圖、純前端怎麼用幾何光線投射算照度、怎麼判斷一條光線有沒有被擋住,以及它在課堂上能怎麼用。可以直接[線上體驗](https://henrychao521.github.io/shadowless-lamp-sim/),也能安裝成 PWA 離線跑。

## 為什麼想做這個

無影燈是個很好的教材,因為它同時踩在兩個我想讓學生理解的概念上。

第一個是**光學的精確表達**。學生(甚至很多大人)直覺會以為「無影燈 = 沒有影子的燈」,好像它用了什麼魔法消滅了影子。但物理上這是不可能的:**只要有不透明的東西擋在光源和檯面之間,那條光路就一定被截斷,該處就一定收不到那顆光源的光**。無影燈做的不是消滅影子,而是**稀釋**影子。「稀釋」和「消滅」這兩個詞的差別,正好是一個很好的科學概念精確化的練習。

第二個是**工程設計的取捨(trade-off)**。要把影子稀釋到肉眼幾乎看不見,工程上有不只一種做法——傳統的多面反射式燈罩、現代的 LED 陣列式,各有優缺點。沒有一種設計樣樣完美,這正是生活科技課想傳達的核心:**工程不是找完美解,而是在限制下做取捨**。

但這兩件事光用講的很抽象。如果能讓學生**自己調燈的數量、角度、遮擋物大小,即時看著陰影變淡或變深**,理解就會從「背一句話」變成「看見它發生」。這就是這個模擬器的目的。

## 一張圖看懂:陰影是怎麼被稀釋的

![無影燈原理示意:多角度補光稀釋陰影](/assets/blog/diagram-lamp.svg)

這張圖是整個專案的核心觀念。上方是無影燈的 **LED 陣列**——注意它不是一顆大燈,而是**很多顆獨立的光源,分佈在不同角度**。中間灰色的橢圓是**遮擋物**(醫生的頭或手),下方是**術野**,我們關心的是它正中心、剛好被遮擋物擋住的那一點。

關鍵在這幾條射線:

- **灰色虛線**是被遮擋物**截斷**的光線。正上方那顆燈的光,直直往中心射,被遮擋物整個擋掉——如果只有這一顆燈,中心就是一塊全黑的**本影(umbra)**。
- **黃色實線**是**繞過遮擋物、從側邊補進中心**的光線。最左邊和最右邊的燈,因為角度夠斜,它們的光路從遮擋物旁邊「擦過去」,還是能照到中心那一點。

於是中心雖然失去了正上方那顆燈,卻仍收得到側邊好幾顆燈的光。**原本應該全黑的本影,被側邊光源填補成只剩淡淡的半影(penumbra)**。只要中心仍保有大約一半以上的亮度,人眼就不會覺得那裡「暗」,也就「看不到影子」了。

一句話總結這張圖:**無影燈靠多角度冗餘補光,把本影打散成淡半影——它稀釋陰影,而不是消滅陰影。**

## 技術段落一:把照度當成「能照到它的光源總和」

模擬器是**純前端**的,用 HTML5 Canvas 2D 做**幾何光線投射(ray casting)**,完全在瀏覽器裡跑,不需要後端。

整個計算的核心觀念其實很單純:**術野上任何一點的亮度(照度),就是「所有能照到它的光源」的貢獻總和。** 「能不能照到」是關鍵——如果某顆燈到這一點的直線被遮擋物擋住了,那顆燈對這一點就沒有貢獻。

我們把術野切成一排(2D)或一格格(3D)的取樣點,對每一個取樣點,掃過每一顆光源,累加沒被擋住的那些:

```javascript
// 概念示意,非實際原始碼
// 對術野上單一點 P,累加所有「看得到它」的光源照度
function illuminanceAt(P, lights, occluder) {
  let total = 0;
  for (const light of lights) {
    // visibility test:這顆燈到 P 的直線有沒有被遮擋物擋住?
    if (isVisible(light.pos, P, occluder)) {
      // 點光源:照度大致隨距離平方衰減(可再乘入射角餘弦)
      const d2 = distSquared(light.pos, P);
      total += light.intensity / d2;
    }
    // 被擋住的光源:對這一點貢獻 0
  }
  return total;
}
```

把術野上每一點都算過一遍,就得到整條剖面(2D)或整片平面(3D)的**照度分佈**。被單一遮擋物擋住的點,因為還有其他光源補光,照度不會掉到 0——這就是半影在數學上的樣子:**不是「有光/無光」的二元,而是「收到幾分之幾的光」的連續值。**

> 光學提醒:真實照度還受**入射角**影響(光斜射時,同樣的光通量攤在更大的面積上,單位面積照度變低,也就是 Lambert 餘弦律)。示意程式裡用距離平方衰減先抓出主結構,要更精確可再乘上入射角的餘弦。原理方向是一致的:**累加可見光源的貢獻。**

## 技術段落二:一條光線到底有沒有被擋住?

上面那段最關鍵的子問題,是 `isVisible()`:**從光源到術野這一點的直線,有沒有穿過遮擋物?** 這在 2D 幾何裡就是一個經典的**線段相交(segment intersection)測試**。

把「光源 → 取樣點」當成一條線段,把遮擋物的邊界當成另一條(或多條)線段。如果兩條線段相交,代表光路被切斷,這顆光源對這一點不可見:

```javascript
// 概念示意,非實際原始碼
// 線段 AB(光源→取樣點)是否與線段 CD(遮擋物的一條邊)相交
function segmentsIntersect(A, B, C, D) {
  const d1 = cross(D.x - C.x, D.y - C.y, A.x - C.x, A.y - C.y);
  const d2 = cross(D.x - C.x, D.y - C.y, B.x - C.x, B.y - C.y);
  const d3 = cross(B.x - A.x, B.y - A.y, C.x - A.x, C.y - A.y);
  const d4 = cross(B.x - A.x, B.y - A.y, D.x - A.x, D.y - A.y);
  // 兩端點分別落在對方線段的兩側 → 相交
  return (d1 * d2 < 0) && (d3 * d4 < 0);
}

function isVisible(lightPos, P, occluder) {
  // 只要光路被遮擋物「任何一條邊」擋住,就不可見
  for (const edge of occluder.edges) {
    if (segmentsIntersect(lightPos, P, edge.a, edge.b)) return false;
  }
  return true; // 暢通無阻 → 這顆燈照得到 P
}
```

這也正是介面上 **2D 幾何光路追跡**畫面在做的事:它把每顆 LED 光源往術野投出許多射線,用上面的測試把它們分成兩類——**暢通的有效光線**畫一種顏色、**被截斷的光線**畫另一種,於是「哪些光被擋掉、哪些光繞了進來」就一目了然。介面裡的「**光路逆行演示**」則是反過來,從術野上的點往回追,看它「收得到哪幾顆燈」,直觀解釋為什麼某一點是本影、某一點只是半影。

## 技術段落三:用「未被遮擋的光源比例」量化本影 vs 半影

有了可見性測試,要區分本影和半影就很自然了。對術野上的一點,定義一個**可見度比例**:

```javascript
// 概念示意,非實際原始碼
// 這一點「看得到幾顆燈 / 總共幾顆燈」
function shadowGrade(P, lights, occluder) {
  let visible = 0;
  for (const light of lights) {
    if (isVisible(light.pos, P, occluder)) visible++;
  }
  const ratio = visible / lights.length;

  if (ratio === 0) return "umbra";       // 一顆都照不到 → 本影(全黑)
  if (ratio < 1)  return "penumbra";     // 部分照得到 → 半影(變淡)
  return "fullyLit";                     // 全部照得到 → 完全照亮
}
```

`ratio` 是 0 就是**本影**——所有光源都被擋住,理論上全黑;介於 0 和 1 之間就是**半影**——只丟失了部分光源,亮度被稀釋但沒歸零;等於 1 就是完全照亮。把這個比例(或實際累加的照度比例)畫成顏色深淺,就得到陰影的濃淡分佈。**3D 版**進一步把整片術野的照度算成**熱圖(照度分佈圖)**,讓學生一眼看出哪裡較暗、哪裡被補滿。

下面這張表把兩種陰影講清楚:

| 項目 | 本影 Umbra | 半影 Penumbra |
|------|-----------|---------------|
| 定義 | 所有光源都被遮擋物擋住的區域 | 部分光源被擋、部分仍照得到的區域 |
| 可見光源比例 | 0 | 介於 0 與 1 之間 |
| 照度 | 理論上趨近 0(全黑) | 保留一部分(被稀釋,非歸零) |
| 邊緣 | 邊界銳利、清楚 | 邊界模糊、漸層 |
| 無影燈的目標 | 盡量讓它**不存在**(消去本影) | 讓它**夠淡**(中心仍保留約一半以上亮度) |

而把這些參數調一調,半影的濃淡就會跟著變,規律也很直觀:

| 你調的參數 | 往哪個方向調 | 陰影的變化 | 為什麼 |
|------------|-------------|-----------|--------|
| 燈的數量 | 增加 | 半影更淡、本影更難形成 | 補光的角度更多,被擋一顆還有更多顆頂上 |
| 光源的角度分散度 | 更分散 | 半影更淡 | 側邊大角度的光更容易繞過遮擋物 |
| 遮擋物大小 | 變大 | 陰影變濃 | 同時擋掉更多光源,本影更容易出現 |
| 遮擋物高度 | 更靠近術野 | 陰影邊緣更銳利、更濃 | 越靠近檯面,能繞過它的角度越少 |
| 燈的安裝高度 | 更高 | 整體較均勻、但中心照度下降 | 距離平方衰減,需在亮度與均勻間取捨 |

最後一列剛好點到工程的**取捨**:燈裝得越高,各角度的光越接近平行、陰影越均勻,但照度也隨距離平方下降。**沒有一個設定樣樣最好**——這正是「LED 陣列式 vs 多面反射式」兩種設計各有取捨的縮影,也是把這個模擬接回工程設計探究的好切入點。

## 醫療標準與離線使用

無影燈不是隨便做做就好,它是醫療器材,對照度與去影能力有國際規範——相關標準是 **IEC 60601-2-41**(手術燈與診斷燈的特定安全要求)。模擬器裡有標註這一點,是想讓學生理解:**課堂上玩的這個光學現象,背後對應的是真實產品要通過的工程驗證。**

工程上,前面那句「中心仍保留約一半以上亮度」並不是隨口說的——去影能力本來就是用「遮擋物在不在時,術野中心照度還剩多少百分比」這類指標來衡量的。模擬器把這件事視覺化,讓抽象的標準變得摸得到。

技術上,整個專案是**純靜態的前端網頁**,再加上 **PWA(Progressive Web App)** 設定——也就是一個 `manifest` 加上 `service worker`,把頁面資源快取下來。第一次連線載入後,之後**沒有網路也能開啟使用**,還能「安裝」到桌面或主畫面像 App 一樣啟動。對網路不穩或乾脆沒網路的教室來說,這點很實用:

```javascript
// 概念示意,非實際原始碼:service worker 把核心資源快取下來
self.addEventListener("install", (e) => {
  e.waitUntil(
    caches.open("lamp-sim-v1").then((c) =>
      c.addAll(["./", "./index.html", "./app.js", "./style.css"])
    )
  );
});
// 之後的請求優先吃快取 → 離線也能跑
self.addEventListener("fetch", (e) => {
  e.respondWith(caches.match(e.request).then((r) => r || fetch(e.request)));
});
```

## 課堂怎麼用(對應 108 課綱)

這個模擬器設計成「**學習目標 / 試試看(先猜再看) / 自我檢核**」三段,本來就是照探究流程鋪的,可以直接搬進課堂:

- **光與照明(光的直進、影子、照度)**:從「光沿直線前進 → 被擋就有影子」出發,用模擬器讓學生看見**本影與半影**的差別,理解「無影燈是稀釋而非消滅影子」。先讓學生猜「怎樣調才能讓影子最淡」,再自己調燈數與角度驗證——**先猜再看**比直接給答案更有效。
- **設計與製作 / 工程的取捨**:用「LED 陣列式 vs 多面反射式」對照,帶學生討論「為什麼沒有一種設計樣樣完美」。調高度那一列(亮度 vs 均勻)就是一個現成的 trade-off 案例,可延伸成設計決策的討論。
- **科技與生活**:從一盞開刀房的燈,看見背後有光學、有幾何、有醫療標準(IEC 60601-2-41)、有工程驗證。讓學生體會**日常醫療器材背後的工程含金量**。
- **科學概念的精確表達**:抓住「稀釋 vs 消滅」這個用詞差異,練習把直覺的說法修正成物理上正確的描述——這是很好的科學素養訓練。
