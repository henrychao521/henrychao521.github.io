把分散在各處的台灣即時開放資料，整合成**一個畫面就能掃完的即時監控儀表板**：左上角是中央氣象署的即時天氣，旁邊是地圖上跳動的地震圓圈，下面是消防 119 的派遣紀錄，再切一頁就能看到全台鐵列車此刻跑到哪裡。整個畫面每分鐘自動刷新，右上角還有個倒數秒數的時鐘。

![Living Tech Portal 台北即時資訊看板：天氣、地震、消防 119、即時頭條](/assets/blog/portal-dashboard.png)

這篇把整個系統從頭到尾拆給你看：資料從哪些開放 API 來、為什麼一定要在中間放一層後端、Cloudflare Tunnel 在做什麼、前端 18 個面板怎麼定時刷新，以及它在生活科技課堂上能怎麼用。

## 為什麼想做這個

學生每天都在用即時資料：早上看天氣 App 決定要不要帶傘、用公車動態 App 抓發車時間、地震速報直接跳到手機上。但他們幾乎不會意識到，這些東西背後其實是**一個個公開的 API**——政府和機構把資料用標準格式對外開放，任何人都能取用。

問題是這些資料**散得到處都是**：天氣要找氣象署、火車要找交通部 TDX、地震美國有 USGS、新聞各家媒體又各自有 RSS。每個來源的格式、更新頻率、認證方式都不一樣。對學生來說，「開放資料」是個抽象名詞；但如果能讓他們**在一個畫面上同時看到這些即時資料一起跳動**，再回頭問「這格天氣是從哪來的？多久更新一次？」，這個抽象就被打通了。

於是就有了這個專案：一個「即時監控中心」，把分散的台灣即時開放資料聚合成統一入口，順便當成**開放資料與 API 串接的活教材**。

## 整體架構一張圖看懂

![Living Tech Portal 開放資料聚合架構](/assets/blog/diagram-portal.svg)

整個系統其實就三層：**左邊是一堆開放資料來源**、**中間是一層自己寫的後端負責聚合與快取**、**右邊是前端的看板式 UI**。看似簡單，但中間那層後端是整個設計的關鍵——下面會解釋為什麼前端不能直接去打那些開放 API。

## 技術段落一：為什麼一定要在中間放一層後端

最直覺的作法是讓前端 JavaScript 直接 `fetch` 各個開放 API。但實務上這條路會立刻撞牆，原因有三個：

1. **CORS（跨來源資源共用）**：瀏覽器基於安全策略，不允許網頁直接抓「別的網域」的資料，除非對方伺服器明確允許。很多開放 API 並沒有開放給瀏覽器直連。
2. **速率限制與金鑰外洩**：像 TDX 這類服務需要 API 金鑰，金鑰若寫在前端 JavaScript 裡，等於攤在所有人眼前；而且每個使用者開一次頁面就直接打一次外部 API，很快就會撞到速率上限。
3. **格式各異**：氣象署、TDX、USGS 吐回來的 JSON 結構天差地遠，前端要為每一種來源寫一套解析邏輯，又難維護。

解法是在中間放一層**後端聚合層**。前端只跟自己的後端講話，後端去跟外面那些 API 周旋，把結果**正規化成統一格式**再吐回來。我用 **FastAPI**（Python 的非同步 Web 框架）來做這層：

```python
# 概念示意，非實際原始碼
from fastapi import FastAPI
import httpx

app = FastAPI()

@app.get("/api/weather")
async def weather():
    # 後端代為呼叫氣象署，金鑰留在伺服器、不外洩給前端
    async with httpx.AsyncClient() as client:
        raw = await client.get(CWA_ENDPOINT, params={"Authorization": CWA_KEY})
    # 把氣象署龐雜的結構，正規化成前端好用的簡潔格式
    return normalize_weather(raw.json())
```

這樣前端拿到的永遠是乾淨、一致的資料，CORS 由後端統一處理掉，金鑰也安全地留在伺服器端。

## 技術段落二：用快取避免每次都打外部 API

光是聚合還不夠。假設有 30 個人同時開著看板，每分鐘刷新一次，如果每次刷新都讓後端真的去打一次氣象署，那一分鐘就是 30 次外部請求——既慢、又浪費、還容易撞速率限制。但氣象署的觀測資料其實**幾分鐘才更新一次**，根本不需要每次都重打。

所以後端要加一層**快取**：第一個人來要資料時去打外部 API 並把結果存起來，接下來短時間內所有人都直接拿這份存好的，等快取過期了才重新去打一次。一個簡單的記憶體快取概念像這樣：

```python
# 概念示意，非實際原始碼
import time

_cache = {}                       # { 來源名稱: (時間戳, 資料) }

async def cached_fetch(key, fetch_fn, ttl=60):
    now = time.time()
    if key in _cache:
        ts, data = _cache[key]
        if now - ts < ttl:        # 還沒過期，直接回快取
            return data
    data = await fetch_fn()       # 過期了才真的去打外部 API
    _cache[key] = (now, data)
    return data
```

`ttl`（存活時間）就是這層設計的旋鈕：天氣可以放長一點（資料本來就慢），地震、消防派遣這種要即時的就放短一點。**選擇合適的 TTL，其實就是在「即時性」和「對外部 API 的禮貌」之間取平衡**——這是個很好的課堂討論題。

## 技術段落三：Cloudflare Tunnel 把本機服務安全地對外

後端寫好了，要怎麼讓網路上的人連得到？傳統作法是租一台有公開 IP 的雲端主機、開防火牆的埠、設定反向代理、處理 HTTPS 憑證——對個人專案來說很麻煩，而且把家裡或學校內網的機器直接對外開埠也有安全顧慮。

我用的是 **Cloudflare Tunnel**：在本機跑一個輕量的 `cloudflared` 連線程式，它會**主動**向 Cloudflare 建立一條加密通道，外面的人連到網域時，流量經由 Cloudflare 從這條既有通道送進來。好處是——**完全不用對外開任何防火牆埠**，本機 IP 也不會曝光，HTTPS 憑證由 Cloudflare 處理掉：

```bash
# 概念示意：把本機 8000 埠的 FastAPI 服務，
# 透過具名通道對應到自己的網域
cloudflared tunnel run living-portal
```

對自架部署來說這是很常見且安全的作法：服務跑在自己掌控的機器上，但對外只露出一個乾淨的網域，沒有開埠、沒有曝露內網。

## 技術段落四：前端 18 個面板的定時輪詢與地圖

前端是**看板式（dashboard）UI**：上方一個下拉選單把 18 個面板分組——總覽（指揮總覽）、城市（台北看板）、交通、環境、太空。每一格面板就是一個獨立的卡片，自己負責去後端要自己那份資料、自己刷新。

刷新用的是最樸實的**定時輪詢（polling）**：用 `setInterval` 每隔固定時間就 `fetch` 一次後端，拿到新資料就重畫畫面。整個看板每分鐘刷新一次，所以畫面上才有那個倒數秒數的時鐘：

```javascript
// 概念示意，非實際原始碼
async function refreshWeather() {
  const data = await fetch('/api/weather').then(r => r.json());
  renderWeatherPanel(data);        // 用拿到的資料重畫天氣面板
}

refreshWeather();                  // 進頁先抓一次
setInterval(refreshWeather, 60_000); // 之後每 60 秒刷新一次
```

地圖類的面板（地震、台鐵路網）用 **Leaflet**——一個輕量的開源地圖函式庫。地震面板會把每一筆地震在地圖上以**圓圈**標示（圓圈大小可對應規模），還能在氣象署與 USGS 兩個來源之間切換；台鐵面板則把**全台鐵道路網與車站**畫在地圖上，再疊上即時列車位置。

台鐵這頁我多做了一點教學味的東西：除了即時位置，還整理了 **6 條主題觀光路線**（平溪線、集集線、南迴看海、花東縱谷、三義舊山線、內灣線），標出沿途看點、顯示「此刻列車正經過哪些風景」，甚至有個「我在哪一站」用定位幫你找最近的車站。

## 18 個面板與它們的資料來源

整個看板用下拉選單分成五大組，這是它的全貌：

| 分組 | 面板 | 主要資料來源 |
|------|------|------------|
| 總覽 | 指揮總覽 | 聚合下列各面板 |
| 城市 | 台北看板（天氣 / 地震 / 消防 119 / 即時頭條） | CWA 氣象署、USGS、新聞 RSS |
| 交通 | 台鐵即時、高鐵即時、航機追蹤、機場看板、YouBike、國道路況、停車場 | TDX 運輸資料流通服務 |
| 環境 | 空品、海象等 | 開放環境資料 |
| 太空 | 太空站、極光、火箭發射 | 公開太空資料 |

幾個關鍵的開放資料來源與它們在這個專案裡的角色：

| 開放資料來源 | 提供單位 | 在本專案的用途 |
|------------|---------|--------------|
| TDX 運輸資料流通服務 | 交通部 | 台鐵 / 高鐵 / 航班 / YouBike / 國道 / 停車場等交通即時資料 |
| CWA 開放資料 | 中央氣象署 | 即時天氣（溫度 / 體感 / 降雨機率 / 紫外線）與地震資訊 |
| USGS Earthquake | 美國地質調查局 | 地震資料（可與氣象署切換對照） |
| 各家新聞 RSS | 多家媒體 | 即時頭條，含發布時間 |

值得一提的是**台北看板裡的消防 119 派遣面板**：它會即時列出救護與火災的派遣紀錄，包含到達現場、離開現場的時間、地址，以及即時的出勤件數——這類資料平常很少有機會這樣攤開來看，放在課堂上談「公共安全資料」很有感。

## 課堂怎麼用（對應 108 課綱）

這個專案幾乎是為「資料蒐集與處理」這個主題量身打造的真實案例：

- **開放資料（Open Data）的取得與整合**：從一個看得到的成品出發，回頭追問每一格資料「從哪個 API 來、多久更新一次、為什麼是公開的」，讓「開放資料」從名詞變成可操作的概念。
- **API 串接與資訊應用**：用 FastAPI 那層當教材，講清楚為什麼前端不能直接打外部 API（CORS、金鑰、速率限制），這是真實系統設計裡天天要面對的取捨。
- **即時系統與資料視覺化**：用「快取 TTL 怎麼選」帶出「即時其實很難」的討論；用 Leaflet 把資料畫到地圖上，示範資料視覺化如何讓抽象數字變得可讀。
- **情境布置與跨域**：做防災、交通、環境主題時，直接投在大螢幕當即時背景；台鐵的主題觀光路線還能延伸到在地與旅遊的跨域討論。

從「感測資料 → 公開 API → 後端聚合 → 前端視覺化」這條完整鏈路，學生能具體看到資料在真實世界裡是怎麼一路被處理、傳遞、呈現的。
